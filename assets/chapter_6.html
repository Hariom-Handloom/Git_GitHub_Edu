üß† Why Learn Git Logs?

Git logs are your project history. Every change, who made it, when, and why ‚Äî all stored in Git. This is essential for:

  • Debugging
  • Collaboration
  • Documentation
  • Reverting mistakes

üîç Basic Command to View Logs

Use this command:

git log

This shows all commits in reverse chronological order (newest first). Each commit includes:

  • Commit hash: Unique ID for each snapshot
  • Author: Who made the commit
  • Date: When the change was committed
  • Message: Description of the change
Example output:

commit d4f7c3a9b234e0f7cabc1a9e98d2f47bfa34a9b1
Author: Jane Doe <jane@example.com>
Date: Fri May 16 14:30:00 2025

Added notes.md with project documentation

üî¢ Show Fewer Commits

To display only the most recent commits, use:

git log -n 3

Replace 3 with any number to limit the list.

üìÉ One-line Summary Format

For a more concise overview:

git log --oneline

Example output:

d4f7c3a Add notes.md with documentation
c2a8f89 Update hello.txt with second line
a3b5d91 Initial commit

This is useful when you just want to see what was changed, quickly.

üìÖ Add Date & Time (Formatted View)

Use:

git log --pretty=format:"%h - %an, %ar : %s"

This shows:

  • Abbreviated hash
  • Author name
  • How long ago
  • Commit message
Example:

d4f7c3a - Jane, 2 hours ago : Add notes.md with documentation

üîÅ View Changes with Each Commit

To see what exactly was changed:

git log -p

This includes a diff after each commit showing lines added/removed.
You can also limit this:

git log -p -n 1

üìÇ View Changes to a Specific File

To see history of just one file:

git log filename.txt

Example:

git log hello.txt

To see changes made to the file:

git log -p hello.txt

üìå Check Who Made Each Line Change (Blame)

Use the blame command to see who last edited each line of a file:

git blame filename.txt

Output format:

a3b5d91 (Jane Doe 2025-05-16) Hello, Git!

This is useful when tracking down bugs or understanding code ownership.

üß† Best Practices

  • Use git log frequently to stay aware of project changes
  • Keep commit messages descriptive so logs are useful
